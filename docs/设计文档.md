## 1. 项目目标

开发一款**纯语音控制**的绘图工具：用户全程不使用鼠标或键盘，仅通过语音指令完成绘图创作。

设计上重点考量四个维度（对应赛题）：

1. **指令理解的准确性与容错性** —— 中文同音/近音纠错、别名归一、模糊匹配、澄清机制。
2. **语音到绘图的响应延迟** —— 本地规则引擎走「快路径」，常见指令毫秒级响应。
3. **复杂指令的拆解与执行** —— 连词切分为原子操作、对象指代解析、（可选）LLM 兜底。
4. **纯语音交互的健壮性** —— 语音模式状态机防止讲解/犹豫被误识别为命令。

## 2. 总体架构

### 双模式设计

为避免「纯前端」与「需要后端」的冲突，采用双模式：

| 模式 | 依赖 | 能力 |
| --- | --- | --- |
| **核心模式**（默认） | 纯前端，零外部依赖，双击 `index.html` 即运行 | 本地规则引擎完成 80%–90% 常见指令，主要演示可独立完成 |
| **增强模式**（可选） | 额外启动 Node 后端 + 自备 LLM API key | 仅用于复杂自然语言指令的 LLM 兜底；关闭后主功能不受影响 |

> 设计取向：本地规则引擎是主角，LLM 仅作兜底，避免「理解能力外包」。

### 数据流

```
麦克风
  → Web Speech API (zh-CN, 流式识别)
  → 中文文本纠错（同音词典 + 中文数字转换）
  → 规则引擎解析  ──命中──→ 指令队列 → Canvas 绘图引擎 → 渲染 + 语音/视觉反馈
                  └─未命中─→ [增强模式] Node 代理 → LLM → 操作 JSON ──┘
```

### 目录结构

```
Chating-Painting/
├── frontend/            # 核心模式：原生 HTML + JS + Canvas，零构建
│   ├── index.html
│   ├── css/style.css
│   └── js/
│       ├── config.js    # 全局配置
│       └── main.js      # 入口：画布初始化与状态显示
├── backend/             # 增强模式：Node LLM 代理（后续 PR 接入）
├── docs/
│   └── 设计文档.md       # 本文件
└── README.md
```

## 3. 语音模式状态机

纯语音应用最大的风险是把用户的讲解、犹豫、口误当成命令。为此设计状态机：

```
        「开始绘图」            「暂停绘图」
 IDLE ──────────────→ LISTENING ⇄──────────────→ PAUSED
                          │      「继续绘图」
              系统播报时   │ ↺ SPEAKING（瞬态）
              暂停识别，   │
              播报完恢复   │
```

- **IDLE**：未启动，不接收任何绘图命令。
- **LISTENING**：监听并执行绘图命令。
- **PAUSED**：暂停执行（用于讲解、思考），命令不生效；说「继续绘图」恢复。
- **SPEAKING**（瞬态）：系统 TTS 播报时暂停识别，避免把自己的播报当成新命令（防自我触发）；播报结束自动回到 LISTENING。
- **「撤销刚才」**：快速纠错，任意时刻可用。

## 4. 对象引用模型

复杂指令的真正难点不是拆句，而是「它 / 那个 / 右边那个 / 第二个圆」指代谁。
为此每个图形对象统一存储：

| 字段 | 含义 | 支撑的指代方式 |
| --- | --- | --- |
| `id` | 唯一标识 | — |
| `type` | 圆 / 矩形 / 线 / 文字 | 「那个圆」 |
| `color` | 颜色 | 「红色的圆」 |
| `x, y, w, h` | 位置与大小 | 「最右边的方块」「上面那个」 |
| `createdOrder` | 创建顺序 | 「第二个红色圆」 |
| `selected` | 是否选中 | 「它」「刚才那个」 |

指代解析命中 0 个或多个时 → 触发澄清机制（见第 5 节）。

## 5. 确认与澄清机制（容错）

不止「没听清，请重说」，而是分情形处理：

- **破坏性二次确认**：「清空画布」→「确定要清空吗？请说确认或取消」。
- **指代不明澄清**：「删掉那个」存在多个候选 →「你是指红色圆形还是蓝色矩形？」。
- **前置条件缺失提示**：「放大一点」但无选中对象 →「请先指定要放大的图形」。
- 进入「等待澄清」子状态时，只接收 确认 / 取消 / 序号 这类应答。

## 6. 计划支持的指令能力

下表为**完整规划**，每项的实现状态会随 PR 推进更新（见第 7、8 节）。

| 类别 | 示例指令 |
| --- | --- |
| 模式控制 | 开始绘图 / 暂停绘图 / 继续绘图 / 撤销刚才 |
| 创建图形 | 画一个红色的圆 / 画矩形 / 画一条直线 / 写字「你好」 |
| 颜色与属性 | 改成蓝色 / 大一点 / 线条粗一点 / 填充 / 只描边 |
| 位置 | 画在左上角 / 画在中间 / 在那个圆的右边 |
| 选择 | 选中红色的圆 / 选第二个 |
| 编辑 | 把圆向右移 / 删掉方块 / 放大它 / 复制一份 |
| 画布 | 撤销 / 重做 / 清空画布（二次确认）/ 保存为图片 |
| 复合指令 | 画三个并排的红圆，然后在最右边画个蓝方块 |
| 帮助 | 我能说什么 |

### 中文容错策略

- **同音/近音纠错词典**：如 `话/划 → 画`、`圆形/圆圈/圈 → 圆`、`长方形 → 矩形`、`篮/兰 → 蓝`。
- **中文数字转换**：`三个 → 3`、`两 → 2`。
- **别名归一表**：颜色、形状、方位的多种说法映射到统一内部表示。
- **模糊匹配兜底**：基于编辑距离对未命中词做近似匹配。
- 识别原文实时显示在屏幕，便于用户察觉误识别。

## 9. 已知限制与风险

- **浏览器兼容性**：Web Speech API 的 `SpeechRecognition` 在 MDN 标注为
  *limited availability*，并非所有主流浏览器都稳定支持中文流式识别。
  **推荐使用 Chrome / Edge 打开**；其他浏览器可能无法启用语音识别。
  - 参考：MDN [SpeechRecognition](https://developer.mozilla.org/docs/Web/API/SpeechRecognition)、
    [Web Speech API](https://developer.mozilla.org/docs/Web/API/Web_Speech_API)。
- **需要麦克风权限**，且多数浏览器要求 HTTPS 或 `localhost` 环境下才允许使用语音识别。
- **首次需一次点击授权**：浏览器安全策略要求开启麦克风必须有一次用户手势，
  因此应用启动时有一个「点击开启」遮罩。这是唯一的手动操作，点击后全程语音控制，
  不违背「纯语音绘图」的核心要求（手势仅用于授权，不参与任何绘图操作）。
- **增强模式需自备 LLM API key**，且依赖网络；核心模式不受影响。
- **环境噪音与口音**会影响识别准确率，故设计了纠错与澄清机制作为补偿。
